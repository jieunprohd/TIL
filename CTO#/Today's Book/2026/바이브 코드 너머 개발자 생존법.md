# 1, 바이브 코딩이란 무엇인가?

AI를 활용하는 방식으로는 즉흥적이고 실험적인 방식인 바이브 코딩과, 명확하고 체계적인 AI 보조 엔지니어링으로 구분할 수 있습니다.

## 1.1 AI 코딩의 스펙트럼: 바이브 코딩부터 AI 보조 엔지니어링까지
### 바이브 코딩
바이브 코딩은 LLM을 통해 코드 생성 과정의 복잡한 작업을 대신 수행하는 동안 사용자는 높은 수준의 목표에 집중하도록 합니다.
모든 함수 작성과 버그 해결을 직접 손으로 작성하는 대신, 자연어로 요청합니다.
먼저 기능을 구상하고 제안 내용을 검토하며, AI가 제공한 결과물을 토대로 여러번 보완해 나가는 방식입니다.

기존 개발자들은 철저히 계획을 세우고, 컴퓨터에게 지시 사항을 상세하게 작성했다면 이제는 오케스트레이션을 통한 결과 도출이 가능합니다.

바이브 코딩은 최소한의 노력으로 바로 사용할 수 있는 해결책을 마련합니다.

**바이브 코딩이 가능해진 요인**
- 최신 AI 코딩 어시스턴트의 발전 -> 코드 생성 뿐만 아니라 수정에서도 뛰어난 성능을 보임
- 코딩 워크플로우에 모델을 원활하게 통합하는 개발자 툴 등작
- AI 어시스턴트에 대한 신뢰도 상승

**바이브 코딩의 장점**
- 생선성 향상

**바이브 코딩의 주의사항**
- 지나친 의존은 다수의 버그나 부실한 설계 결정으로 이어질 수 있음
- 초기 계획 단계를 중요하게 여기지 않는 경향 -> 초기 작업 생략으로 명혹한 방향 없는 산만한 개발로 이어질 수 있음

**바이브 코딩이 적합한 케이스**
- 데모를 직접 제작하는 단독 개발자인 케이스
- 낮은 진입 장벽으로 AI로 프로덕트를 개발해보고 싶은 케이스
- 뜻밖의 창의적인 결과를 기대하는 케이스

** 바이브 코딩의 결과물은 시작점이지, 최종 결과물이 아님을 인지

### AI 보조 엔지니어링
AI 보조 엔지니어링은 빠르게 새로운 기능을 구현하거나 현재 존재하는 기능을 개선하는 과정에서 각 단계마다 AI를 조력자로 활용합니다.
바이브 코딩은 프롬프트를 우선으로 생각하며 최종 결정까지 한다면, AI 보조 엔지니어링은 계획을 우선으로 삼고 개발자가 개발 과정을 주도하며 최종 결정을 내립니다.
AI 보조 엔지니어링은 엔지니어를 대체하는 것이 아닌 제약 조건을 준수하여 창의력을 발휘하며 보완하여 효율을 높입니다.

**바이브 코딩과 AI 보조 엔지니어링의 목적**
바이브 코딩의 목적은 *단기적인 속도와 작동하는 코드를 빠르게 작성*에 가깝다면, AI 보조 엔지니어링의 목적은 *지속 가능한 속도와 신뢰성, 고품질의 코드를 효율적으로 작성*이라고 볼 수 있습니다.

**AI 보조 엔지니어링이 적합한 케이스**
- 이미 체계적인 프로세스가 있고, 그 체계를 유지하면서 개발하려는 케이스
- 명세가 명확하게 존재하는 모듈을 구현하는 케이스

### 두 방식 중 하나를 선택하는 것이 아닌 결합하는 것
프로젝트를 진행할 때, 초기에는 바이브 코딩을 통해 새로운 토대를 구성하고, 이후 어느 정도 체계가 갖추어진 후에는 AI 보조 엔지니어링 원칙에 따라 체계적으로 개발을 진행할 수도 있습니다.
바이브 코딩은 기존 경로를 벗어나 신속하게 새로운 가능성을 탐색하도록 도와주며, 창의적인 시도를 할 수 있습니다.
AI 보조 엔지니어링은 정해진 궤도 위를 달리는 안정성이 보장된 방식입니다.

AI로 개발을 지속하다보면 AI가 잘하는 일과 못하는 일을 구분하게 되고, 문제 단위로 세분화하여 AI에게 개별적으로 제공하는 방법을 익혀 오케스트레이션 방식으로 진화할 수 있습니다.

## 1.2 코드 라인을 넘어서: 의도 중심 프로그래밍
기존의 프로그래밍은 원하는 목적지로 가기 위해 단계별로 지시를 내렸다면, 이제는 목적지를 전달하면 AI가 스스로 최적의 경로를 찾아서 갈 수 있습니다.
앞으로 개발자는 **어떻게**에서 **무엇**에 집중하여, 보다 수준 높은 추상화로 작업할 수 있습니다.

### 프롬프트
프롬프트는 AI 코딩 시스템에 입력하거나 질문하는 텍스트를 의미합니다. 

**동작 방식**
1. 프롬프트 이해 -> 프롬프트를 입력하면 텍스트를 분석하고 통계적 패턴을 활용해 의도를 추론, 이를 통해 합리적인 코드 완성 결과를 예측
2. 컨텍스트 활용 -> 프롬프트 정보에만 의존하는 것이 아닌 다른 정보들을 함께 반영해 종합적인 판단
3. 코드 생성 -> 모델이 의도나 취선의 추정을 마친 후 코드 생성, 프롬프트와 컨텍스트를 명확하게 입력하면 수준 높은 퀄리티의 결과를 얻을 수 있음
4. 인간 감독을 통한 검증 -> 코드의 정확성과 프로젝트의 요구사항 충족 등에 대한 최종 결정은 개발자가 내려야 함

**장점**
- 개발자의 생산성 향상
- 개발자이ㅡ 집중력 유지
- 진입 장벽 축소
- 개발자의 역할과 기술의 변화 -> 무엇과 왜에 집중하며 아키텍처 설계, 문제 정의, 검증과 같은 기술이 중요해짐
- 생산성, 창의성 -> AI가 반복 작업을 처리하면서 그 시간에는 고수준의 창의적 작업에 집중할 수 있음

**단점**
- 신뢰와 정확성
- 저수준 기술의 망각
- 취업 시장의 변화

### 바이브 코딩의 이상적인 사용 사례
- 제로 투 원 프로덕트 개발 -> 완전히 새로운 것을 개발할 때 빠르게 MVP를 생성할 수 있음
- 기능 프로토타이핑 및 CRUD 어플리케이션 -> 반복적이면서 시간 소모가 큰 CRUD 작업을 AI에게 맡길 수 있음
- 코드 통합 -> 언어를 변경하거나 형식을 변경하는 경우 AI가 빠르게 처리할 수 있음
- 현대 프레임워크 활용
- 반복적인 코드 생성

### AI 보조 엔지니어링이 우선인 상황
- 구현에 정확한 제어가 필요한 경우 (알고리즘 구현, 계산 문제 해결 등)
- 미션 크리티컬 시스템 개발 초기

** 바이브 코딩과 AI 보조 엔지니어링 사이의 **전환 시점**

### AI가 여전히 어려움을 겪는 영역
- 매우 복잡한 시스템 
- 저수준 최적화 및 시스템 프로그래밍
- 독특하거나 생소한 프레임워크
- 창의적인 UI/UX 디자인
- 의도와 요구사항 해석

### 고려해야할 제한 사항과 트레이드오프
- 변수 출력 품질
- 프롬프트의 모호함 -> 코드도 모호해짐
- 콰도한 의존에 따른 기술 퇴화
- 프라이버시와 보안 문제
- AI 결과의 편향
- 인간적 요소와 신뢰
