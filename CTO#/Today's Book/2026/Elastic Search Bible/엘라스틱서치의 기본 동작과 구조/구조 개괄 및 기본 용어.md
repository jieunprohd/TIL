## 엘라스틱 서치 구조 개괄 및 기본 용어

![image.png](attachment:117ea714-d609-4d76-854e-f65020f75200:image.png)

- **node**: es 프로세스 하나가 노드 하나를 구성 → 노드 하나가 여러개의 샤드를 가지는 구조지만, 고가용성을 위해 같은 종류의 샤드를 같은 노드에는 배치하지 않음
- **cluster**: es 노드 여러 개가 모여 하나의 클러스터를 구성
- **shard**: 여러 샤드로 분리하여 분산 저장하는 역할, 고가용성을 제공하기 위해 주 샤드와 복제본 샤드로 구성됨
- **document**: es가 저장하고 색인을 생성하는 JSON 문서 (:= ROW)
- **index**: 문서를 모아 놓은 단위 (:= TABLE)

노드의 개수보다 샤드의 개수가 많아도 상관없음 왜? 샤드는 클러스터가 알아서 분배할 것이기 때문에

## 엘라스틱서치 내부 구조와 루씬

루씬은 문서를 색인하고 검색하는 라이브러리

루씬은 색인한 정보를 파일로 저장하기 때문에 검색을 하기 위해서는 파일을 열어야하고, 파일을 연 시점에 색인이 완료된 문서만 검색할 수 있음 → 이후 변경사항 발생 시 파일을 새로 열어야 함

### 루씬 flush

1. 문서 → 인메모리 buffer에 쌓임
2. flush 발생
3. buffer 내용을 새로운 segment 파일로 생성

결과 → _inverted index segment 파일 생성됨, 검색 가능 (디스크 저장 X)

### 루씬 commit

1. commit 시점까지 생성된 segment들 모음
2. 디스크에 fsync (완전히 기록)
3. segments_N 파일 생성 → 가장 최신 segments_N 파일 기준으로 전체 상태 판단

결과 → 디스크 저장

**왜 fsync를 사용할까?**

fsync는 OS는 해당 파일의 page cache 내용을 물리 디스크에 모두 기록하고, 완료될 때까지 호출한 쪽을 블로킹하기 때문에 fsync 성공 후 commit point 기록이나 translog 정리를 해야 안전함

### segment

불변인 데이터로 구성되어 있는 루씬의 검색 대상 → 불변인 세그먼트 개수를 무작정 늘릴 수 없어 적당히 병합 수행

- **생성** → 새로운 세그먼트 생성
- **삭제** → 세그먼트에 삭제 플래그만 표시
- **업데이트** → 삭제 플래그 표시 후 새 세그먼트 생성

### 루씬 인덱스와 엘라스틱서치 인덱스

여러개의 segment들이 모여 하나의 루씬 인덱스를 이루고, 루씬은 **해당 인덱스 내에서만 검색이 가능**

es shard가 하나의 루씬 인덱스를 wrap한 구조임

**es index**

- 문서 저장 요청이 들어올 시 해당 내용을 라우팅 하여 여러 샤드에 분산시켜 저장, 색인
- 검색 요청이 들어올 시 각 샤드를 대상으로 검색을 한 후 결과를 병합하여 응답 → **루씬에서 불가능했던 분산 저장, 검색을 가능하게 함**

### translog

es에 색인된 문서들은 commit이 되어야 디스크에 안전하게 기록, 매 요청마다 commit은 비용이 많이 들지만 장애 발생시 commit 되지 않은 데이터는 날라갈 수 있음

translog는 인메모리 버퍼 저장 시점에 생기고, 디스크 저장 시점에는 정리됨 → 백그라운드에서 주기적으로 flush를 실행해 적절한 크기 유지

commit 이전에 장애 발생 시 segments_N과 translog를 비교해 날라간 데이터를 복구함
