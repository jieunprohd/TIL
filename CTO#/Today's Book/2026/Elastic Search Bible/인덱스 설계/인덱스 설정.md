## number_of_shard

인덱스가 데이터를 몇 개의 샤드로 쪼갤 것인지 결정하는 설정 → 한 번 지정하면 변경이 어려움 (reindex등의 특별한 작업 필요)

**샤드 개수가 미치는 영향**

- 많은 경우 → 클러스터의 성능과 색인 성능이 감소 (샤드 하나마다 루씬 인덱스 하나씩 더 생성, primary shard 하나당 replica shard도 늘어남)
- 적은 경우 → 샤드 하나의 크기가 너무 커지면서 장애 발생 시 샤드 복구가 오래 걸리고 안정성이 떨어짐

## number_of_replicas

주 샤드 하나당 복제본 샤드의 개수를 결정하는 설정

number_of_replicas는 쉽게 수정이 가능함

```bash
PUT [인덱스 이름]/_settings

{
  "index.number_of_replicas": [변경할 개수]
}

# 0으로 설정할 경우 복제본 X, 주 샤드만 가지고 있음
# 보통 bulk insert 시 성능 향상을 위해 0으로 설정했다가 원복
```

## refresh_interval

해당 인덱스의 refresh 주기를 결정하는 설정 (기본적으로는 1s 설정 → 1초마다 refresh, -1은 주기적으로 refresh 수행 X)

index.search.idle.after → n초 동안 검색 쿼리가 들어오지 않는 경우 다음 검색 쿼리가 들어올 때가지 refresh를 수행하지 않고 대기

## 인덱스 설정을 지정하여 인덱스 생성

```bash
# 1. 기본 설정으로 인덱스 생성
PUT [인덱스 이름]

**response**
{
    "acknowledged": true, #타임아웃 전 클러스터 상태 업데이트 성공 여부 -> false여도 인덱스 생성 아예 실패 X, 시간이 지난 후 정상적으로 생성 가능
    "shards_acknowledged": true, # 타임아웃 이전에 지정한 개수만큼 샤드 활성화 성공 여부
    "index": [인덱스 이름]
}

# 2. 인덱스 삭제
DELETE [인덱스 이름]

# 3. 인덱스 설정을 지정하여 인덱스 생성
PUT [인덱스 이름]

{
    "settings": {
      [인덱스 설정]
    }
}
```
