# 매핑과 필드 타입

## 동적 매핑 vs 명시적 매핑

**동적 매핑** → 인덱스 필드의 타입을 지정하지 않고 es에서 자동으로 매핑하는 방식, 예상하지 못한 신규 필드 추가시 유연 운영 가능

**명시적 매핑** → 사용자가 직접 매핑을 지정해주는 방식, 예정된 신규 필드 추가 시 명시적 지정 추천

** 필드 타입 및 매핑 설정 대부분은 한 번 지정되면 사실상 변경이 불가능하기 때문에 신중해야함

```bash
PUT [인덱스 이름]/_mapping

{
    "properties": {
      [추가할 필드의 타입과 정보]
    }
}
```

**dynamic 설정 옵션 / index / copy_to**

```bash
PUT [인덱스 이름]

{
    "mappings": {
        "dynamic": "strict", # 기본적으로는 true
        "properties": {
            "first_name": {
                "type": "text",
                "copy_to": "full_name",
                "index": false
            },
            "last_name": {
                "type": "text",
                "copy_to": "full_name",
                "index": false
            },
            "full_name": {
                "type": "text",
                "index": true
            }
        }
    }
}
```

**dynamic 설정 옵션에 따른 동작 방식**

- **true** → 새 필드를 자동으로 매핑에 추가하고 색인
- **false** → 새 필드를 색인하지는 않지만 _source에 저장, 검색 불가
- **strict** → 새 필드가 들어오면 예외 발생 및 문서 색인 거부

**index 필드 사용 예시**

정렬, 집계, 스크립드 동작에서만 사용되고, 검색 대상은 아닌 필드에 유용 (doc_values만 유지, index는 미사용)

es 버전 8.1부터는 false로 설정해도 필드가 doc_values를 사용할 경우 doc_values를 통해 검색이 됨 → 성능이 많이 떨어짐

**copy_to**

여러 필드의 값을 하나의 가상 필드로 복사하여 통합 검색을 가능하게 하는 파라미터 → 역색인에만 존재, 문서 조회시에는 가상 필드이기 때문에 X

## 필드 타입

### 심플 타입

#### 숫자 타입

- long → 64비트 부호 있는 정수
- integer → 32비트 부호 있는 정수
- short → 16비트 부호 있는 정수
- byte → 8비트 부호 있는 정수
- double → 64비트 부동소수점 수
- float → 32비트 부동소수점 수
- half_float → 16비트 부동소수점 수
- scaled_float → double 고정 환산 계수로 스케일링하여 long으로 저장되는 부동소수점 수

** 작은 비트를 사용할 수록 검색과 색인 시 성능이 향상되지만, 저장시에는 실제 값에 맞춰 최적화되기 때문에 디스크 사용량과는 무관함

**scaled_float 타입이란?**

환산 계수의 인자가 100이고, 실제 값이 3.14라면 314로 색인 → 검색, 집계, 정렬 시에는 3.14로 동작

→ 정확도는 떨어지나, 압축이 쉬워 디스크 공간 절약이 가능

#### date 타입

date 타입에서 사용 가능한 포맷 종류

- epoch_millis → 밀리초 단위로 표현한 epoch 시간
- epoch_second → 초 단위로 표현한 epoch 시간
- date_time → yyyyMMdd 형태
- strict_date_time → yyyy-MM-dd’T’HH:mm:ss.SSSZZ 형태
- date_optional_time → yyyy-MM-dd 또는 yyyy-MM-dd’T’HH:mm:ss.SSSZ 형태
- strict_date_optional_time → 위와 동일하나 연, 월, 일이 4자리, 2자리, 2자리 보장

** 여러 포맷을 사용하고 싶은 경우 `“format”: “포맷 || 포맷”`으로 이어 붙이면 됨

#### 배열

es는 별도의 배열을 위한 타입 구분이 없어 지정한 필드 타입에 맞는 한 가지 종류의 데이터 배열을 넣을 수 있음

es 색인 과정에서 데이터가 배열이라도 각각 값에 대해서 역색인 구성

### 계증 구조를 지원하는 타입

#### object 타입

json 데이터를 넣을 경우 기본적으로 object 타입으로 생성이 됨

**enabled 설정**

```bash
PUT [인덱스 이름]/_mappings

{
    "properties": {
        "notEnabled": {
            "type": "object",
            "enabled": false
        }
    }
}
```

doc_values를 생성하지 않고, 파싱도 하지 않아 성능상 이득을 볼 수 있음

es가 파싱조차 수행하지 않음

→ _source 제외 아무 곳에도 저장되지 않음 → 검색, 정렬, 집계 x

→ object 타입이 아닌 데이터가 들어와도 타입 충돌 x

→ 같은 종류의 타입이 아닌 배열이 들어와도 저장 가능

```bash
PUT [인덱스 이름]/_doc

{
    "spec": [
            {
            "cores": 12,
            "memory": 128,
            "storage": 8080
        },
        {
            "cores": 6,
            "memory": 64,
            "storage": 8080
        },
        {
            "cores": 6,
            "memory": 32,
            "storage": 4000
        }
    ]
}
```

**object 타입 저장 방식**

```bash
{
    "spec.cores": [12, 6, 6],
    "spec.memory": [128, 64, 32],
    "spec.storage": [8080, 8080, 4000]
}
```

**문제**

- 객체들 사이의 경계가 사라짐 → 검색 시 `cores: 6, memory: 128`인 데이터는 없기 때문에 검색이 되지 않아야 하지만, 평탄화 작업으로 인해 검색 결과에 포함되게 됨

#### nested 타입

object 타입과 다르게 배열 내 각 객체를 독립적으로 취급

```bash
PUT [인덱스 이름]
    {
        "mappings": {
            "properties": {
                "spec": {
                    "type": "nested", # 기본적으로 객체는 object 타입으로 생성되기 때문에 명시적 설정 필요
                    "properties": {
                        "cores": {
                        "type": "long",
                    },
                    "memoery": {
                      "type": "long",
                    },
                    "storage": {
                        "type": "long",
                    },
                }
            }
        }
    }
}
```

**nested 타입 저장 방식**

![image.png](/image/nested-architecture.png)

자식에 대한 검색 시 hidden doc 기반으로 검색 후 parentBitSet 기반으로 부모 doc을 join하여 응답 반환

동작 방식은 특수하기 때문에 nested 전용 쿼리(nested 쿼리 지정 후 기존 검색 쿼리 사용)를 이용해서 검색해야함

```bash
POST [인덱스 이름]/_search

{
    "query": {
        "nested": {
            "path": "spec", # nested 쿼리는 path를 기준으로 어느 hidden document 그룹을 탐색할지 결정
            "query": {
                "bool": {
                    "must": [
                        { "term": { "spec.cores": 6 } },
                        { "term": { "spec.memory": 64 } }
                    ]
                }
            },
        }
    }
}
```

- 검색 시 nested 쿼리를 사용해야 컨텍스트가 hidden doc으로 전환되어 자식 객체 기반으로 검색
- nested 미사용 시 내부 bool 쿼리가 아무리 `spec.cores`를 참조해도 부모 doc의 평탄화된 값 (hidden doc 관련 필드 X) 기반 검색

nested는 각 객체를 별도 문서로 분리해 저장하기 때문에 성능 문제가 발생할 수 있어 제한 존재

1. index.mapping.nested_fields.limit → 한 인덱스에서 nested 타입 최대 지정 가능 개수 (기본 50개)
2. index.mapping.nested_objects.limit → 한 문서에서 nested 객체 최대 개수 (기본 10000개)

### 특수 타입

- geo_point → 위도, 경도 저장하는 타입
- geo_shape →  지도상 특정 지점에 선, 도형 등을 표현하는 타입
- binary → base64로 인코딩된 문자열을 저장하는 타입
- long_range, date_range, ip_range … → 수, 날짜, IP 등의 법위를 저장하는 타입
- completion → 자동완성 검색을 위한 특수한 타입

**completion 타입의 저장 방식**

FST(Finite State Transducer) 자료구조 기반 데이터 저장

**FST 자료구조**

![image.png](/image/fst-architecture.png)

공통 prefix를 가진 단어들이 노드를 공유하는 방식

일반 inverted index의 경우 토큰 전체를 키로 저장해 prefix 기준 조회 시 전체 토큰을 다 조회해야 하지만, FST는 prefix 기반으로 트리를 타고 내려가면 돼서 prefix 길이만큼의 탐색으로 모든 후보를 바로 열거할 수 있음

#### text vs keyword

- **text** → 애널라이저가 적용된 후 색인되는 타입, 일치 검색에 적합
- **keyword** → 토큰 단위로 쪼개지 않고 색인되는 타입 (노멀라이저 적용, 간단한 전처리), 전문 검색에 적합

** term은 토큰에 후처리 작업 후 최종적으로 역색인에 들어가는 형태

**analyzer 기본 동작 방식**

1. **character filter** → 토큰화 이전에 원본 텍스트 자체를 변환 (ex. HTML 태그 제거, 특정 문자 치환)
2. **tokenizer** → 텍스트를 토큰으로 분리 (ex. 공백 기준, 형태소 기준)
3. **token filter** → 토큰화 이후 각 토큰을 가공 (ex. 소문자 변환, 불용어 제거, 동의어 처리)

→ 커스텀 애널라이저를 생성하여 적용 가능

**Analyzer의 text 전처리 과정**

1. **Hello, World!** 문자열 입력
2. standard 애널라이저 전처리를 거쳐 **[hello, world]** 토큰으로 쪼개져 역색인
3. 이후 **hello**라는 값에 대해서 검색해도 조회가 가능

**Normalizer의 keyword 전처리 과정**

1. **Hello, World!** 문자열 입력
2. 특정 노멀라이저를 적용하도록 지정하지 않는 경우 **Hello, World!**로 역색인
3. 이후 **Hello, World!** 로 정확히 검색하는 경우에만 검색 가능

**multi-fields**

```bash
PUT [인덱스 이름]

{
    "mappings": {
        "properties": {
            "last_name": {
                "type": "text", # 전문 검색용 기본 필드
                "copy_to": "full_name",
                "index": false,
                "fields": {
                    "keyword": {
                      "type": "keyword" # 서브 필드
                    }
                }
            }
        }
    }
}
```

last_name의 경우 애널라이저가 적용된 전문 검색용 필드

last_name.keyword → 토큰화 없이 정렬, 집계, 완저 일치 검색 등에 사용

**대부분의 경우 text를 기본 필드, keyword를 서브 필드로 두는 이유**

**text**는 analyzer를 통과하면서 원본 문자열이 term으로 존재하지 않으니 정렬, 집계, 정확한 일치 검색이 안 됨

**keyword**는 analyzer를 거치지 않아 정확한 일치 검색, 정렬, 집계가 되지만, 전문 검색은 불가능

보통 두 가지를 동시에 필요로 하는 경우가 많아 사용자가 특정 단어로 검색할 수도 있고, 완전 일치로 집계하거나 정렬을 할 수도 있어 multi-fields로 관리함

### doc_values

검색은 `term → doc` 방향이라 빠르지만, 정렬·집계·스크립트 작업은 `doc → 값` 방향으로 읽어야 함
이를 위해 `doc ID → 값` 방향으로 미리 정렬해 디스크에 저장해두는 것이 doc_values → 정렬, 집계, 스크립트 성능 향상
text, annotated_text를 제외한 대부분의 타입에서 지원
정렬·집계·스크립트를 거의 사용하지 않는 필드라면 doc_values를 false로 설정해 디스크 공간을 절약할 수 있음

### fielddata

text 필드는 analyzer로 토큰화되기 때문에 doc_values를 생성할 수 없음 → 필요 시 런타임에 JVM 힙 메모리에 구성
역색인 전체를 힙에 올리는 방식이라 메모리 부담이 크고 OOM(Out of Memory) 문제가 발생할 수 있어 기본값은 false

### _source

문서 색인 시점에 전달된 원본 JSON 저장 필드 → 역색인이 되지 않아 검색 대상 X

원본 JSON을 그대로 사용하기 때문에 디스크 사용량이 많음 → enabled 설정을 통해 저장하지 않을 수도 있음

```bash
PUT [인덱스 이름]

{
    "mappings": {
        "_source": {
          "enabled": false
        }
    }
}
```

**_source 비활성화 시 생길 수 있는 문제**

1. 업데이트와 update_by_query 사용 불가 → segment는 불변이기 때문에 기존 문서를 삭제하고 새로 생성해야하는데 기존 문서가 없어 불가능
2. reindex 사용 불가 → 원본이 없어 사용 불가능

**업데이트와 update_by_query 사용 불가 이유**

es의 segment는 불변으로, 수정 시 기존 문서를 제거하고 재생성

→ 원본 문서를 저장하지 않음으로 수정 불가

**reindex란?**

원본 인덱스에서 내용을 대상 인덱스에 새로 색인하는 작업

** es는 메이저 버전이 하나 차아나는 인덱스만 읽어들일 수 있어, 메이저 버전 업그레이드 전 reindex를 수행해 인덱스를 재생성해야함

### 인덱스 코덱 변경

디스크 공간 절약을 위해 성능을 희생할 수 있다면 데이터 압축률을 높이는 편이 나음

```bash
PUT [인덱스 이름]

{
    "settings": {
        "index": {
          "codec": "best_compression"
        }
    }
}
```

#### **codec 설정 값**

**default** → LZ4 압축

압축률보다는 압축 및 해제 속도에 최우선 순위를 둔 압축 방식

**best_compression** → DEFLATE 압축

### synthetic source

_source에 JSON 원문을 저장하지 않고, _source가 필요한 경우 doc_values에서 데이터를 재조립하여 동작하는 방식

**특징**

1. 인덱스의 모든 필드가 doc_values를 사용 (text, annotated_text x)
2. _source를 읽어야 하는 작업의 성능이 떨어짐
3. 재조립 시 원문 JSON과 다를 수 있음 (필드 이름이나 배열 내의 값은 오름차순으로 재정렬)
4. 인덱스의 크기를 많이 줄여줌

```bash
PUT [인덱스 이름]

{
    "mappings": {
        "_source": {
          "mode": "synthetic"
        }
    }
}
```
