### 검색 엔진

다른 RDBMS나 경쟁사 NoSQL에 비해 매우 강력한 검색 기능 지원

단순 텍스트 매칭부터 전문 검색 지원, 역색인을 통한 검색 속도 향상, 애널라이저를 조합하여 색인 구성 가능

es는 기술적으로는 NoSQL 문서 저장소이기도 하지만, 본질적 목적은 검색 엔진, RDBMS나 순수 NoSQL보다 검색 성능이 압도적으로 뛰어남

** **역색인** → 일반 색인은 문서에서 단어를 찾지만, 역색인은 단어에서 문서를 찾도록 구성됨
미리 단어가 어떤 문서에 있는지 매핑해두어, 검색 시 전체 문서를 스캔하지 않고 찾음

** **애널라이저** → 텍스트 데이터를 토큰 단위로 분류하고 가공하여 역색인에 저장하는 핵심 기능

redis는 인메모리 캐시, mongoDB는 범용 문서 DB, es는 검색 특화 엔진

→ 검색 기능만 놓고 보면 ES가 압도적이지만, 트랜잭션이나 유연한 쿼리가 필요하다면 MongoDB나 RDBMS가 적합

### 분산 처리

데이터를 여러 노드에 분산 저장 → 검색이나 집계 작업 수행시에도 분산 작업 지원

es 클러스터는 여러 개의 노드로 구성되고, 데이터는 샤드 단위로 분할되어 여러 노드에 분산 저장됨

검색, 집계 요청도 각 노드가 병렬로 처리한 후 결과를 합쳐서 반환하는 방식

** **클러스터** → 하나의 논리적인 es 인스턴스 (여러개의 노드들이 모여 하나의 es 시스템처럼 동작하는 단위)

** **노드** → 클러스터를 구성하는 개별 서버
- **Master Node** → 클러스터 상태 관리, 샤드 할당 결정, 노드 추가/삭제 관리, **데이터 추가 X**
- **Data Node** → 실제 데이터 저장, 검색/집계 처리
- **Coordinating Node** → 요청을 받아 적절한 노드로 분산, 결과를 모아서 반환
- 기타 → Ingest Node (전처리 노드), ML Node (머신러닝 노드) …

** **샤드** → 데이터를 나누어 저장하는 최소 단위
- **Primary Shard** → 원본 데이터 저장, 인덱스 생성 시 개수 고정
- **Replica Shard** → primary shard 복제본, 장애 대응용, 1개 이상 추천
→ 샤드가 너무 많은 경우 오버헤드 발생 가능, 너무 적은 경우 병렬 처리 불가 → 10 ~ 50GB 권장

**데이터 저장 시**

1. client 요청
2. Coordinating Node에서 요청을 받고, 어떤 샤드에 저장할 지 결정
3. 해당 Primary Shard에서 데이터 저장
4. Replica Shard에서 복제본 저장

**데이터 검색 시**

1. client 요청
2. 모든 관련 샤드에 쿼리 전파
3. 각 샤드에서 결과 반환
4. Coordinating Node에서 결과를 모아서 반환

### 고가용성

es는 항상 설정된 replica count를 유지하려고 함 → 노드가 죽으면 replica count만큼 복제본을 재생성하여 개수 유지

복제본이 많을 수록 읽기 성능과 가용성은 증가하지만, 쓰기(색인) 성능이 떨어질 수 있음

### 수평적 확장성

es는 새 노드를 클러스터에 추가하기만 하면 자동으로 데이터 재분배와 샤드 이동이 이루어져 무중단 확장이 가능

새 노드 추가 시 es에서 자동적으로 샤드 리밸런싱을 수행하여 새 노드에도 데이터가 균등하게 분배됨

** **수직적 확장** → 기존 서버의 CPU / 메모리를 향상 시키는 방식
**수평적 확장** → 서버를 추가하는 방식

### JSON 기반의 REST API 제공

JSON 형태의 문서를 저장, 색인, 검색함

es에 작업 요청을 보낼 떄도 JSON 기반의 REST API 사용

### 데이터 안정성

일부 시스템은 응답을 먼저 보내고 디스크에 비동기로 쓰는 경우가 있음

es는 색인 요청에 대해서 디스크 기록이 완료된 후 200 OK를 보내기 때문에 응답 시점에 데이터 유실 위험이 없음

### 다양한 플러그인을 통한 기능 확장

### 준실시간 (near real-time search) 검색

데이터를 색인 직후 바로 검색하면 실패할 수 있음 → 내부적으로 역색인이 완료되어야 함

**역색인 과정**

1. memory buffer → 메모리에 데이터 저장, translog 기록
2. refresh → segment 생성 (검색 가능)
3. flush → 디스크에 영구 저장 commit, translog 정리, commit point 생성, segment merge

보통 1초 정도 걸림 → 따라서 완전한 실시간은 아니고 **준실시간**임

바로 검색되게 하기 위해서는 `refresh_interval -1` 설정 또는 API 요청 시 `?refresh=true` 파라미터 추가 가능 → 색인 성능이 떨어짐

bulk insert 시 `refresh_interval -1`, `replica count 0` 설정 후 원복

![image.png](attachment:64989378-158e-4716-b132-be81d5efae30:image.png)

**merge를 하는 이유**

- refresh 시 segment들이 생기고, es는 검색을 위해 모든 segment들을 돌아야 해 검색 속도가 느려짐
- 문서 삭제 시 delete 마킹만 하는 방식(실제 데이터 제거 X)으로 메모리 낭비

→ merge는 cpu와 디스크 I/O를 꽤 쓰기 때문에 백그라운드 스레드에서 실행하고 `merge throttling`으로 속도를 제한해, 색인·검색 성능에 미치는 영향을 줄임

**Tiered Merge Policy**

같은 tier(크기 범위)의 segment가 10개 이상 쌓이면 → 자동 merge 트리거, merge 후 결과 segment는 상위 tier로 올라가 다시 같은 규칙 적용

### 트랜잭션 미지원

RDBMS와 다르게 트랜잭션 미지원

정합성을 지키기 위해서 es는 단독 원본 DB로는 사용하지 않고, RDBMS에 원본 저장, es는 검색용 동기화로 사용

개별 문서 단위로 색인시 성공/실패 각각 처리, bulk 사용시 일부 문서가 실패해도 나머지는 성공

### JOIN 미지원

join이라는 데이터 타입은 존재하나 제한적인 상황을 위한 기능이며 성능도 떨어짐

RDBMS와 다르게 데이터를 **비정규화**해 저장이 필요

** es의 join 타입 → 하나의 index 안에서 Parent-Child를 같이 저장하는 방식

** es는 테이블 단위로 나누는 것이 아닌 검색에 필요한 모든 정보를 하나의 index로 구성
