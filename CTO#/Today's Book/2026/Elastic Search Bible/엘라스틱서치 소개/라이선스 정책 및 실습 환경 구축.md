## 실습 환경 구축

설치 시 설정 사항

```yaml
clsuster.name: test-es # 클러스터 이름 설정
node.name: test-es-node01 # 노드 이름 지정, 각 노드의 이름은 클러스터 내에서 고유해야함

path:
data: /ojieun/elasticsearch/data # 데이터를 저장할 경로 (2개 이상 가능)
logs: /ojieun/elasticsearch/logs # 로그를 저장할 경로

network.host: 127.0.0.1 # es와 연결될 네트워크 주소

discovery.type: "single-node" # 단일 노드의 개발 모드로 동작, 다른 노드의 클러스터 참가 신청 무시
xpack.security.enabled: false # es에서 제공하는 보안 관련 기능을 사용할 것인지 지정
```

실무에서 es를 운영하기 위해서는 최소 3대 이상의 노드를 클러스터로 구성이 필요 → 고가용성, 안정성, 성능

**2대일 때 발생할 수 있는 문제**

- **Split-Brain** → 네트워크 장애로 인해 서로 통신이 되지 않을 경우 각자 자신이 Master Node라고 판단하여 클러스터가 분리되는 현상
- **과부하** → 하나의 노드가 죽었을 때 다른 노드에 부하가 집중되어 성능이 저하되거나 서비스가 멈출 수 있음

**최소 3대 이상 노드를 사용하는 이유**

- **Split-Brain 해결** → 투표 시스템을 통해 과반수를 획득한 노드가 정상적인 클러스터로 인정되어, 1대가 죽더라도 클러스터 안정적으로 운영 가능
- **과부화 분산** → 하나의 노드가 죽어도 남은 두 대의 노드가 부하를 분산하여 처리할 수 있음
- **역할 분담** → 노드가 많을 수록 Master Node, Data Node, Coordinating Node의 역할이 분담되어 효율적인 자원 활용이 가능

```yaml
-Xms2g # 시스템 메모리의 절반으로 지정 또는
-Xmx2g # JVM의 Compressed OOPs 기능이 적용되도록 하는 경곗값을 찾아 지정하는 것이 좋음
```

**JVM의 Compressed OOPs (Ordinary Object Pointer)란?**

64bit JVM의 경우 객체 참조 하나당 8바이트 사용 → 메모리 낭비가 심함

64bit의 주소를 32bit로 줄여서 저장하는 방식 → 항상 8의 배수로 저장되기 때문에 하위 3bit를 버려서 저장

**왜 31g를 추천할까?** → JVM에서는 힙의 크기가 32GB 이하일 때만 compressed oops 사용, 32GB 넘을 시 기존 8바이트 단위 저장

**2g일 경우 문제**

1. 힙의 old 부분이 금방 차면서 gc가 자주 돌게 됨 → stw 발생 및 요청 지연 빈번
2. es 캐시 사용 불가 → 캐시 공간이 없어 계속 디스크를 읽게 되면서 검색 속도가 크게 늘어남
