## 템플릿

유사한 구조의 인덱스를 자주 생성 할 때 사전 정의한 설정대로 생성할 수 있도록 설정

### 인덱스 템플릿

동일한 템플릿의 인덱스를 여러개 생성할 때 사용

```bash
PUT _index_template/[인덱스 템플릿 이름]

{
    "index_patterns": [
      "[부합할 시 적용되는 이름, *은 와일드카드]"
    ],
    "priority": 1, # 값이 클 수록 템플릿 간 우선순위가 높아짐
    "template": {
        "settings": {
            "number_of_shards": 2,
            "number_of_replicas": 1
        },
        "mappings": {
            "properties": {
              ...
            }
        }
    }
}
```

`index_patterns`와 일치하는 이름의 인덱스가 생성되는 경우 자동으로 템플릿 적용

### 컴포넌트 템플릿

템플릿 간 중복되는 부분 묶어 생성할 때 사용

```bash
PUT _component_template/[컴포넌트 템플릿 이름]

{
"template": {
    "mappings": {
        "properties": {
          ...
        }
    }
}

{
    "template": {
        "settings": {
            "number_of_shards": 2,
            "number_of_replicas": 1
        }
    }
}
```

```bash
PUT _index_template/[인덱스 템플릿 이름]

{
    "index_templates": [""],
    "composed_of": ["컴포넌트 템플릿 이름"]
}
```

`composed_of`에 적용시킬 컴포넌트 템플릿 이름을 넣으면 재사용이 가능

### 레거시 템플릿

위 인덱스 템플릿과 컴포넌트 템플릿은 7.8.0 버전부터 추가된 기능으로, 이전 버전에서 사용하던 템플릿

인덱스 템플릿과 기능은 동일하나, 컴포넌트 템플릿 적용 불가

```bash
PUT _template/[템플릿 이름]

{
    "index_patterns": [],
    "settings": {
    ...
    },
    "mappings": {
    ...
    }
}
```

동일한 `index_patterns`에 대해서 레거시 템플릿의 우선순위가 더 낮음

### 동적 템플릿

새로 들어온 필드의 매핑을 사전에 정의한대로 동적 생성하는 기능

```bash
PUT _index_template/[인덱스 템플릿 이름]

{
    "index_patterns": [],
    "priority": 1,
    "template": {
        "settings": {},
        "mappings": {
            "dynamic_templates": [
                {
                "my_text": { # 동적 템플릿 이름
                    "match_mapping_type": "string", # 동적 템플릿 적용 타입
                    "match": "*_text", # 동적 템플릿 적용 필드명
                    "mapping": {
                      "type": "text" # 매핑시킬 es 타입
                    }
                },
            {
                "my_keyword": {
                    "match_mapping_type": "string",
                    "match": "*_keyword",
                    "mapping": {
                      "type": "keyword"
                    }
                }
            ]
        }
    }
}
```

**동적 템플릿 적용 시 확인시킬 수 있는 조건**

- match_mapping_type → 데이터의 타입을 json 파서를 이용해 확인
- match / unmatch → 필드 이름이 일치하는지, 일치하지 않는지 확인
- path_match / path_unmatch → match / unmatch와 동일하게 동작하지만, 마침표가 들어가는 경우 사용

### 빌트인 인덱스 템플릿

로그나 메트릭을 편리하게 수집하기 위해 `metrics-*-*`과, `logs-*-*` 인덱스 패턴에 priority가 100인 템플릿이 사전 정의 되어 있음

** 해당 인덱스 패턴 형식의 템플릿 사용 시 pritory를 더 높게 생성하는 등의 조치 필요

## 라우팅

document 저장 시 어떤 es shard에 저장할 지 지정할 수 있음

지정하지 않을 시 기본적으로 _id 값에 따라 라우팅

색인 시 라우팅 값 지정 시, 조회, 업데이트, 삭제, 검색 등에서도 동일하게 지정해야 함

```bash
PUT [인덱스 이름]/_doc/[doc_id]?routing=[라우팅 시 사용할 본인의 id]

{
  ...
}
```

라우팅 값을 명시하지 않는 경우 전체 샤드를 대상으로 검색을 수행하고, 명시하는 경우 단일 샤드를 대상으로 검색

인덱스의 크기가 크지 않는 경우 성능의 차이가 크게 없지만, 실제 운영환경에서는 성능 차이가 매우 큼

### 인덱스 내에서의 _id 고유성 보장

라우팅 값을 명시하지 않는 경우 성능이 떨어지고, 검색 결과가 달라질 수 있음

하지만, 라우팅 값이 틀리는 경우 문서가 존재하는 경우에도 검색을 실패할 수 있음

인덱스 내에서의 _id 값의 고유성은 샤드 단위 보장으로, 라우팅 값을 다르게 지정할 경우 한 인덱스 내에서 같은 _id 값으로 여러개의 문서가 생길 수 있음

### 인덱스 매핑에서 라우팅 필수 지정

```bash
PUT [인덱스 이름]

{
    "settings": {
        "_routing": {
          "requires": true / false
        }
    }
}
```

`requires: true`인 경우 이후 `?routing=[_id]` 누락 시 에러 발생
